Overview

The blog engine is a standalone Node.js CLI tool that generates airport parking articles using a multi-step AI pipeline. It scrapes competitor content, generates articles with Claude, quality-checks them, scores SEO, and publishes to Payload CMS.

Content Model

Articles follow a 3-tier hierarchy for topical authority and internal linking:

HUB Pillar Page

Example: "The Complete Guide to JFK Airport Parking"

  • 2,000 – 2,500 words
  • Comprehensive overview of ALL parking aspects
  • Each H2 briefly introduces a subtopic
  • Links out to every sub-pillar article
  • The definitive, authoritative page for the airport

SUB-PILLAR Detailed Guide

Example: "JFK Long-Term Parking: Rates, Tips & Best Lots"

  • 1,500 – 2,000 words
  • Deep dive on one specific aspect
  • Links back to hub + cross-links to sibling sub-pillars
  • More detail than the hub provides

SPOKE Focused Piece

Example: "Is JFK Cell Phone Lot Free?"

  • 800 – 1,200 words
  • Answers one specific question or niche topic
  • Links to parent sub-pillar + hub
  • Targets long-tail keywords

Step 1: Scrape Competitors

scraper.ts

How It Works

  1. Search Google via SerpAPI for the target keyword (top 5 results)
  2. Optionally uses manual competitor URLs from the queue item
  3. For each URL, fetch the page with a 15-second timeout
  4. Parse with Cheerio — strip nav, footer, sidebar, scripts, styles
  5. Extract: title, all H2/H3 headings, body text (capped at 10,000 chars)
  6. 3-second delay between scrapes to avoid rate limits

Output: ScrapedArticle

{
  url: "https://competitor.com/jfk-parking-guide",
  title: "JFK Parking Guide 2026",
  headings: ["How Much Does JFK Parking Cost?", "Best Off-Site Lots", ...],
  content: "JFK airport offers several parking options..."  // max 10,000 chars
}

Step 2: Analyze (AI)

Claude Sonnet 4.5 prompts/analyze.ts

What the AI Does

Reads all scraped competitor articles and identifies patterns, gaps, and opportunities.

Prompt Inputs

  • Target keyword
  • All scraped articles (title + headings + content for each)

AI Output

faqQuestions6-8 high-intent FAQ questions

Key Instruction

The analysis prompt emphasizes entity coverage — the AI is told to identify terminal names, airline hubs, roads, transit options, and neighborhoods mentioned across competitors, since entity density is a strong NLP ranking signal.

Step 3: Write (AI)

Claude Sonnet 4.5 prompts/write.ts

What the AI Does

Generates the full article HTML using the analysis results, airport data, external links database, and 32 writing rules.

Prompt Inputs

From Queue Item

  • Title, keyword, airport code
  • Article type (hub/sub-pillar/spoke)
  • Article style (standard/narrative/etc.)
  • Pre-defined outline (if any)
  • Hub/parent slugs for cross-linking

From Analysis + Data

  • Common topics & content gaps
  • Recommended headings
  • Airport facts (terminals, rates, transit)
  • Verified external links database

Writing Rules (32 total)

Grouped by purpose:

HTML & Structure (Rules 1-4)

  • 1 Allowed tags: h2, h3, p, ul, ol, li, a, strong, em, blockquote, table, thead, tbody, tr, th, td
  • 2 Forbidden: h1, div, span, img, inline styles, classes, IDs
  • 3 No article title in HTML (handled separately by CMS)
  • 4 Internal links: /blog/[slug] format

Links & CTAs (Rules 5, 9-11)

  • 5 External links: Only from verified database or .gov sites. Use rel="nofollow" where specified. Never invent URLs.
  • 9 Reference "Triply" or "TriplyPro" with link to triplypro.com
  • 10 Early CTA (first 500 words): Must be CONTEXTUAL — reference a specific terminal, price, or parking type from surrounding content. Never generic. Links to /search?airport=JFK
  • 11 Closing CTA: Ties back to article's main promise. References specific airport.

Readability (Rules 12-15)

  • 12 Sentence length: Average 15-20 words. Never exceed 30. Mix short (8-12) with medium (20-25).
  • 13 Simple words: "use" not "utilize", "help" not "facilitate". 8th-grade reading level.
  • 14 Active voice: "The shuttle picks you up" not "You will be picked up by the shuttle".
  • 15 No compound sentences: Split multi-clause sentences into two.

AI Search & Featured Snippets (Rules 16-26)

  • 16 Opening answer: First paragraph = concise direct answer, extractable on its own.
  • 17 Key Takeaways: 4-6 bullet list with bold lead phrases, right after opening.
  • 18 Use <ul> / <ol> lists naturally in most sections.
  • 19 Answer-first sections: Each H2 opens with 1-2 sentence direct answer.
  • 20 Question headings: Frame H2s as questions where natural.
  • 21 Definition pattern: "What is X? X is..." for new concepts (not obvious ones).
  • 22 Comparisons: At least one structured comparison section.
  • 23 Source attribution: Cite real sources. NEVER fabricate.
  • 24 E-E-A-T signals: "based on current rates", "travelers frequently report".
  • 25 Bold key terms: Use <strong> for important names and phrases.
  • 26 Entity coverage: Mention terminals, airlines, roads, neighborhoods, transit.

Quality & Freshness (Rules 27-32)

  • 27 Freshness signals: "as of 2026" for time-sensitive data. Don't date evergreen facts.
  • 28 Paragraph length: 3-5 sentences (80-120 words). Never exceed 5 sentences.
  • 29 No filler: Every sentence = a fact, tip, or actionable detail.
  • 30 Comparison tables: Use HTML tables for pricing data and side-by-side comparisons.
  • 31 Verification dates: Add "(verified February 2026)" to time-sensitive claims.
  • 32 PAA targets: Include 2-3 "People Also Ask" style headings.

Step 4: Edit (AI)

Claude Sonnet 4.5 prompts/edit.ts

What the AI Does

A second AI pass that reviews the generated article against 21 quality checks. Fixes issues, improves structure, and assigns a quality score.

Style preservation is critical: The edit prompt includes a preamble for each article style telling the AI what NOT to change. For example, narrative articles should NOT have a Key Takeaways list added, and listicle articles should keep their numbered headings.

21 Quality Checks

Accuracy & Structure (1-7)

  • 1 Remove made-up statistics or prices
  • 2 Keyword in first paragraph, 2+ H2s, conclusion
  • 3 Internal links use /blog/[slug]
  • 3b External links only from approved database
  • 4 Only allowed HTML tags
  • 5 Readability: Flesch-Kincaid grade 6-9
  • 6 Early + closing CTAs present and contextual
  • 7 Remove AI phrases ("In conclusion", "game-changer")

AI Search Optimization (8-18)

  • 8 Opening = direct answer to main query
  • 9 Key Takeaways list exists near top
  • 10 Lists in most H2 sections
  • 11 Answer-first sections (1-2 sentence lead)
  • 12 Question-format headings
  • 13 Source attribution (never fabricated)
  • 14 Bold key terms (2-4 per section)
  • 15 Paragraph & sentence length limits
  • 16 No filler sentences
  • 17 E-E-A-T credibility language
  • 18 Freshness signals on time-sensitive data

New Checks (19-21)

  • 19 Comparison tables: If comparing options/rates, convert to HTML table
  • 20 Verification dates: Time-sensitive claims need "(verified Month Year)"
  • 21 Table presence: Data-heavy/comparison articles must have at least one table

AI Output

{
  "html": "<h2>Edited content...</h2>...",
  "changes": [
    "Shortened opening paragraph for directness",
    "Added Key Takeaways section",
    "Converted pricing list to table",
    "Added verification dates to rate claims"
  ],
  "qualityScore": 87
}

Data Sources

The AI is grounded with verified, factual data to prevent hallucination.

Airport Data

data/airports/JFK.json LGA.json

Verified airport facts injected into the write prompt:

  • Full airport name & authority
  • Terminal list with airlines per terminal
  • Nearby roads, transit options, neighborhoods
  • Official parking rates & shuttle info
  • Distance from city center
  • liveSources — verified URLs grouped by topic (official, transit, parking, lounges, EV, etc.)

Rule: The AI must use ONLY these facts for specific claims. Uncertain facts get hedging language ("typically", "around").

External Links Database

data/external-links.json

50 verified outbound links with metadata:

  • URL, domain, title, description
  • Authority level (high / medium / low)
  • Suggested anchor texts (vary naturally)
  • rel attribute (dofollow / nofollow / sponsored)
  • Relevant airports & article types
  • Categories: official-airport, government, transit, parking-aggregator, lounge-guide, etc.

CMS Publishing

payload.ts html-to-lexical.ts

HTML → Lexical Conversion

Payload CMS stores rich text in Lexical JSON format, not raw HTML. The converter transforms AI-generated HTML into Lexical's node tree:

HTMLLexical NodeNotes
<h2> - <h6>headingWith tag property (h2, h3, etc.)
<p>paragraph
<ul> / <ol>listWith listType: bullet/number
<li>listitemSupports nested lists
<a>linkPreserves href + rel attribute
<strong> / <em>text (format bitmask)Bold=1, Italic=2
<blockquote>quote
<table>tableStores raw HTML for rendering
<br>linebreak

Post Creation

After conversion, a CMS post is created with:

  • title, slug, excerpt
  • content — Lexical JSON document
  • seo — metaTitle + metaDescription
  • faqItems — question/answer pairs
  • category + tags (found or created)
  • featuredImage — uploaded via Unsplash
  • articleType, airportCode, seoScore
  • status: "draft" (requires manual review before publishing)

CLI Commands

Generate Articles

npm run generate                   # Generate next queued article
npm run generate -- -n 5           # Generate 5 articles
npm run generate -- -t hub         # Only hub articles
npm run generate -- -a JFK         # Only JFK airport
npm run generate -- --dry-run      # Preview without creating posts

Publish Batch

npm run publish -- -b batch-1      # Publish all drafts in "batch-1"
npm run publish -- -b batch-1 --dry-run

Score Articles

npm run score -- --slug jfk-airport-parking-deals
npm run score -- --airport JFK     # Score all JFK articles
npm run score -- --all             # Score everything
npm run score -- --all --save      # Score and save back to CMS

Report

npm run report                     # Full queue status
npm run report -- -a JFK           # JFK only
npm run report -- -b batch-1       # Specific batch
